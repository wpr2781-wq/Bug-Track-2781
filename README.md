## Project overview

Bug·Tracker lets a development team log, assign, and track software bugs through their full lifecycle — from discovery to resolution. Each bug is recorded as a **ticket** with a priority level, a status, an assignee, and target dates.

The application runs entirely in the browser. There is no backend. Data survives page refreshes because it is saved to `localStorage` as JSON.

## Data model

All data is stored in three `localStorage` keys. Each key holds a **JSON array** of objects.

### `bt_people`

Represents team members who can report or be assigned to issues.

```json
{
  "id":       "person_seed_1",
  "name":     "Saiesha",
  "surname":  "Naidoo",
  "email":    "sai@gmail.com",
  "username": "sai_n"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique ID generated by `_makeId()` |
| `name` | string | First name |
| `surname` | string | Last name |
| `email` | string | Email address (stored lowercase) |
| `username` | string | Unique handle — enforced in `createPerson()` |

---

### `bt_projects`

Represents software projects that issues are linked to.

```json
{
  "id":   "project_seed_1",
  "name": "Portfolio Website"
}
```

---

### `bt_bugs`

Represents individual bug reports / tickets. This is the main entity.

```json
{
  "id":                   "bug_1714000000000_a3f2c",
  "summary":              "Login page crashes on Safari",
  "description":          "White screen appears after clicking login on Safari 16.",
  "identifiedBy":         "person_seed_1",
  "dateIdentified":       "2026-04-10",
  "projectId":            "project_seed_1",
  "assignedTo":           "person_seed_2",
  "status":               "open",
  "priority":             "high",
  "targetResolutionDate": "2026-04-20",
  "actualResolutionDate": null,
  "resolutionSummary":    ""
}
```

| Field | Type | Allowed values |
|-------|------|----------------|
| `status` | string | `open` · `in-progress` · `resolved` · `closed` |
| `priority` | string | `low` · `medium` · `high` · `critical` |
| `identifiedBy` | string | ID of a person from `bt_people` |
| `assignedTo` | string / null | ID of a person, or `null` if unassigned |
| `projectId` | string / null | ID of a project from `bt_projects` |

## localStorage — how it works

`localStorage` is a browser API that stores **key-value pairs** as strings. The data survives page refreshes and browser restarts but is scoped to the origin (the folder the files are opened from).

### The three operations used in `storage.js`

```js
// Save — converts a JavaScript array to a JSON string and stores it
localStorage.setItem('bt_bugs', JSON.stringify(bugsArray));

// Load — reads the string back and converts it to a JavaScript array
var bugs = JSON.parse(localStorage.getItem('bt_bugs')) || [];

// Delete — removes a key entirely
localStorage.removeItem('bt_bugs');
```

### How `storage.js` wraps these

All pages import `storage.js`, which exposes clean functions so no page needs to call `localStorage` directly:

```js
getAllBugs()            // returns the full array
findBug(id)            // returns one bug or null
createBug(data)        // adds a new bug and saves
updateBug(id, changes) // merges changes into an existing bug and saves
deleteBug(id)          // removes a bug and saves
```

The same pattern is used for `getAllPeople()` / `createPerson()` / `deletePerson()` and `getAllProjects()` / `createProject()` / `deleteProject()`.

## Page-by-page breakdown

### `index.html` — Dashboard

- Calls `getAllBugs()` on load and distributes cards across four Kanban columns: **Backlog**, **Ready**, **In Progress**, **Done**.
- Each card shows the issue summary, a colour-coded priority badge, and the assigned person's username.
- Clicking a card navigates to `view-issue.html?id=<bug_id>`.

### `create-issue.html` — New Issue

- Populates the *Identified By* and *Assign to* dropdowns from `getAllPeople()` and the *Project* dropdown from `getAllProjects()`.
- On submit, calls `createBug(data)` and resets the form.
- Includes a `sample_Bugs()` function that seeds two example bugs if none exist yet.

### `view-issue.html` — View & Edit Issue

- Reads `?id=` from the URL: `new URLSearchParams(window.location.search).get('id')`.
- **Detail view** — renders all fields read-only. Reporter and assignee show with initials avatars. Resolution fields are hidden unless status is `resolved`.
- **Edit form** — toggled by the *Edit Issue* button on the same page. Pre-populates every field. Calls `updateBug(id, changes)` on save and snaps back to the detail view with a toast confirmation.
- **Delete** — confirms with the user, calls `deleteBug(id)`, then redirects to `index.html`.

### `people.html` — People

- Form creates a new person via `createPerson(name, surname, email, username)`.
- `createPerson()` throws an error if the username is already taken — the page catches this and shows an error toast.
- Lists all people in a table with an avatar (initials), username, email, and a remove button.

### `projects.html` — Projects

- Form creates a new project via `createProject(name)`.
- Lists all projects showing how many open issues are linked to each one.
- Shows an *All clear* badge when a project has no unresolved issues.

## Status and priority logic

### Status values

| Value | Meaning |
|-------|---------|
| `open` | Issue has been logged, no work started |
| `in-progress` | Actively being worked on |
| `resolved` | Fix has been applied — resolution date and summary should be filled in |
| `closed` | Issue closed without a fix, or a duplicate |

> When status is set to **resolved** on the edit form, the *Resolution details* section becomes visible and the resolved date is auto-filled with today's date.

### Priority values

| Value | Badge colour | Meaning |
|-------|-------------|---------|
| `low` | Green | Minor issue, no urgency |
| `medium` | Amber | Affects functionality but a workaround exists |
| `high` | Red | Significant impact, needs prompt attention |
| `critical` | Red | System-breaking, must be fixed immediately |

### Auto-overdue logic

On `view-issue.html` load, the page compares the bug's `targetResolutionDate` against today's date. If the ticket is still `open` or `in-progress` and the target date has passed, the status is updated to `overdue` and saved back to localStorage automatically.

```js
function syncOverdue() {
  var today = new Date().toISOString().split('T')[0];
  var bugs  = getAllBugs();
  var changed = false;
  bugs.forEach(function(b) {
    if ((b.status === 'open' || b.status === 'in-progress')
        && b.targetResolutionDate
        && b.targetResolutionDate < today) {
      b.status = 'overdue';
      changed = true;
    }
  });
  if (changed) _saveList('bt_bugs', bugs);
}
```

## Known limitations

- **Single sign-on only** — there is no login system. Anyone with access to the files can use the app. This is by design per the project specification.
- **No relationships enforced** — deleting a person does not automatically unassign them from existing bugs. Their ID remains on the bug record.
- **localStorage is origin-scoped** — data is tied to the exact folder/browser. Opening the files on a different machine or in incognito mode starts fresh.
- **No file upload for avatars** — profile pictures are not implemented; initials are used as a fallback.
